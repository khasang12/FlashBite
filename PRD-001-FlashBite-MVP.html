1. Multi-Tenant Business Vision & SaaS Strategy Business

FlashBite is an enterprise-grade, hyper-local on-demand delivery platform structured as a multi-tenant Software-as-a-Service (SaaS) platform. Instead of operating as a single localized food delivery app, FlashBite enables third-party regional operators, massive restaurant holding groups, or white-label delivery enterprises (each designated as a Tenant) to spin up their own isolated hyper-local marketplaces instantly.

1.1. The B2B Multi-Tenant Value Proposition

Every tenant on the FlashBite engine enjoys total data, customer, and catalog isolation while running on shared, highly dynamic global compute clusters. This business architecture addresses critical B2B market needs:

  • Regional Market Autonomy: An operator in Berlin and an operator in Tokyo run completely customized corporate configurations, localized currencies, independent fee matrices, and isolated driver fleets.
  • Dynamic Cost & Operational Efficiency: Infrastructure overhead scales with tenant transactional volume. Small tenants do not pay for heavy dedicated server instances; their isolation is logically guaranteed at the database, streaming, and state machine layer.
  • Data Sovereignty & Legal Compliance: Tenant data is robustly sandboxed. A customer or order record from Tenant A can never leak or be calculated inside the telemetry dashboards of Tenant B.
Operational Context for Education Purposes

To demonstrate production paradigms within this unified monorepo, everything runs locally under multi-tenant structures. The system acts as if it is managing completely independent corporations simultaneously via unique identifiers propagated across all asynchronous communication nodes.

2. Multi-Tenant Marketplace Dynamics Business

The marketplace mechanics change dramatically once multi-tenancy is applied. The system handles multiple dimensions of isolation across actors within individual domains:

2.1. Tenant Mapping & Isolation Rules

The user experience is separated at the presentation tier using dynamic subdomains handled inside NextJS routing protocols:

  • tokyo-sushi.flashbite.com/... routes to Tenant B's customer experience.
  • admin.flashbite.com manages the global telemetry control grid for platform configuration.

3. Tenant-Isolated Core Persona Journeys Business

The core lifecycle of operations relies completely on passing the immutable identity tenant context across every edge interaction.

3.1. The Customer Checkout Journey

  1. Tenant Handshake: The customer accesses the platform via a tenant-specific URL (e.g., https://alpha-meals.flashbite.com). The NextJS server identifies the tenant context and polls the localized configuration.
  2. Menu Interaction: The customer views item catalogs which are dynamically hydrated via cached read-models specific to that tenant ID.
  3. Command Dispatch: When clicking "Place Order", the frontend dispatches a command payload containing an implicit HTTP header: X-Tenant-ID: alpha-meals-uuid.

3.2. The Restaurant Fulfillment Journey

  1. Isolated Listening: The Merchant Dashboard application connects to the NestJS Event-API via Server-Sent Events (SSE). The connection query parameter contains the verified tenantId.
  2. SLA Monitoring: Incoming order alerts populate. The system tracks individual tenant-defined SLA limits (e.g., Tenant Alpha mandates restaurant responses within 4 minutes, while Tenant Beta allows up to 7 minutes).

3.3. The Driver Delivery Stream

  1. Contextual Dispatch: Drivers receive order dispatch proposals isolated to the tenant fleet they are logged into.
  2. High-Velocity Telemetry Isolation: The driver app streams GPS location coordinates to the server every 3 seconds. Every message is accompanied by the active tenantId payload key, allowing Kafka consumer groups to partition real-time metrics safely.

6. Unified Technical Topology & Architectural Flows Technical

The architectural strategy leverages NestJS as a microservices controller, NextJS for user interfaces, Kafka as the reliable cross-service transactional event freeway, and Temporal for business orchestration state persistence.

6.1. The Global Structural Architecture Flow

The system operates using an isolated dual-path mechanism (CQRS + Event Sourcing + Outbox):

  1. Command Pipeline: NextJS ➔ NestJS Write-API ➔ PostgreSQL Transaction (Event Ledger + Outbox Record inside a single atomicity boundary) ➔ Outbox Poller Worker ➔ Kafka (Avro Encoded Binary Stream with Tenant Header propagation).
  2. Query Pipeline: Kafka Stream ➔ NestJS Projection Worker ➔ MongoDB Tenant Isolated Document Collections ➔ Redis Cache Layer (Namespaced by Tenant ID) ➔ NestJS Read-API ➔ NextJS UI Fetch Component.
  3. Process Orchestration Path: Kafka Consumer Worker ➔ Temporal Workflow (Tracks state timelines, timers, and SLA conditions for specific tenantId configurations).

7. Technical Multi-Tenant Isolation Strategies Technical

To ensure high performance and physical data sandbox compliance within a single project context, we apply explicit boundaries at each technical tier:

7.1. Database Isolation: Row-Level Security (RLS) in PostgreSQL

We leverage a shared-database, isolated-row paradigm using native Postgres policies. Every table contains a tenant_id column indexed robustly alongside primary identifiers. All application queries apply tenant variables implicitly via NestJS transaction middleware wrappers.

7.2. Kafka Isolation: Partitioning & Consumer Groups

We route multi-tenant messages over global shared topics (e.g., order-events), but enforce isolation through the Partition Key strategy. When streaming messages, the Kafka producer sets the Message Key to tenantId:orderId. Kafka uses this key to calculate hashes, ensuring that all messages belonging to a given tenant's order always land on the exact same partition in chronological sequence.

7.3. Temporal Isolation: Multi-Tenant State Scoping

Temporal workflows run deterministically. Multi-tenancy is achieved by injecting the tenantId as a mandatory root workflow execution input argument and setting the WorkflowId structure pattern to tenant_id:order_id. This prevents cross-tenant workflow collision and guarantees trace execution isolation.

7.4. Redis Caching Namespacing

To avoid key collision and data contamination, all keys stored inside the shared Redis instance match an immutable structural pattern: tenant:{tenantId}:{domain}:{resourceId}.

8.1. NestJS Dynamic Tenant Context Middleware

This interceptor extracts the tenant context from the incoming HTTP request headers or subdomain routing structures and injects it into an asynchronous execution context container (AsyncLocalStorage).

// apps/write-api/src/common/tenant-context.middleware.ts
import { Injectable, NestMiddleware, UnauthorizedException } from '@nestjs/common';
import { Request, Response, NextFunction } from 'express';
import { AsyncLocalStorage } from 'async-hooks';

export const tenantStorage = new AsyncLocalStorage<string>();

@Injectable()
export class TenantContextMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    const tenantId = req.headers['x-tenant-id'] as string;
    
    if (!tenantId) {
      throw new UnauthorizedException('Multi-tenant execution bounds require an active Tenant Identifier header.');
    }

    // Wrap execution context inside AsyncLocalStorage
    tenantStorage.run(tenantId, () => {
      req['tenantId'] = tenantId;
      next();
    });
  }
}

8.2. Multi-Tenant Event Store & Transactional Outbox Handler

This command handler ensures that the Event Sourcing ledger and the Transactional Outbox entry are written within an atomic, tenant-bound PostgreSQL database execution context.

// apps/write-api/src/orders/commands/handlers/create-order.handler.ts
import { CommandHandler, ICommandHandler } from '@nestjs/cqrs';
import { PrismaService } from '../../../prisma/prisma.service';
import { tenantStorage } from '../../../common/tenant-context.middleware';

@CommandHandler(CreateOrderCommand)
export class CreateOrderHandler implements ICommandHandler<CreateOrderCommand> {
  constructor(private readonly prisma: PrismaService) {}

  async execute(command: CreateOrderCommand) {
    const tenantId = tenantStorage.getStore();
    const { orderId, customerId, totalAmount, items } = command;

    const orderPayload = { orderId, customerId, totalAmount, items };

    // Atomicity block execution
    return await this.prisma.$transaction(async (tx) => {
      // 1. Write to append-only Event Store ledger
      await tx.eventStore.create({
        data: {
          tenantId: tenantId,
          aggregateType: 'ORDER',
          aggregateId: orderId,
          eventType: 'OrderPlacedEvent',
          payload: orderPayload
        }
      });

      // 2. Append to Transactional Outbox for Kafka relay execution
      await tx.outboxLedger.create({
        data: {
          id: crypto.randomUUID(),
          tenantId: tenantId,
          topic: 'order-events',
          eventType: 'OrderPlaced',
          // Message key pattern preserves partition ordering per tenant constraints
          partitionKey: `${tenantId}:${orderId}`,
          payload: JSON.stringify({ ...orderPayload, tenantId }),
          status: 'PENDING'
        }
      });
    });
  }
}

8.3. Multi-Tenant Temporal Workflow Execution Engine

This workflow maps out how Temporal dynamically coordinates tenant-specific configurations while processing order milestones.

// packages/temporal-backend/src/workflows/multi-tenant-order.workflow.ts
import { proxyActivities, sleep, defineSignal, setHandler } from '@temporalio/workflow';
import type * as activities from '../activities/order.activities';

const { chargePaymentActivity, fetchTenantConfigActivity, refundPaymentActivity } =
  proxyActivities<typeof activities>({ startToCloseTimeout: '2 minutes' });

export const merchantApprovalSignal = defineSignal<[boolean]>('merchantApproval');

export async function tenantOrderLifecycleWorkflow(tenantId: string, orderId: string, orderDetails: any): Promise {
  let isApprovedByMerchant = false;
  let responseReceived = false;

  setHandler(merchantApprovalSignal, (approved: boolean) => {
    isApprovedByMerchant = approved;
    responseReceived = true;
  });

  // 1. Look up localized tenant business SLAs dynamically from metadata storage
  const tenantConfig = await fetchTenantConfigActivity(tenantId);
  const slaDurationString = `${tenantConfig.restaurantAcceptanceTimeoutMinutes} minutes`;

  // 2. Execute Payment Activity
  await chargePaymentActivity(tenantId, orderId, orderDetails.totalAmount);

  // 3. Coordinate Tenant SLA countdown window natively inside Temporal state machine
  const completedInTime = await Promise.race([
    sleep(slaDurationString).then(() => false),
    new Promise<boolean>((resolve) => {
      setHandler(merchantApprovalSignal, (approved) => resolve(approved));
    })
  ]);

  if (!completedInTime || !isApprovedByMerchant) {
    // Invoke Compensation pattern in case of SLA breach
    await refundPaymentActivity(tenantId, orderId, orderDetails.totalAmount);
    return `ORDER_CANCELLED_VIA_SLA_BREACH_FOR_TENANT_${tenantId}`;
  }

  return 'ORDER_PROCESSED_SUCCESSFULLY';
}

8.4. Namespacing Redis Cache within the NestJS Read-API

This service shows how the Read-API abstracts data queries by prefixing keys with the tenant identifier to ensure isolation.

// apps/read-api/src/orders/orders-query.service.ts
import { Injectable, Inject } from '@nestjs/common';
import { CACHE_MANAGER } from '@nestjs/cache-manager';
import { Cache } from 'cache-manager';
import { tenantStorage } from '../common/tenant-context.middleware';

@Injectable()
export class OrdersQueryService {
  constructor(@Inject(CACHE_MANAGER) private redisCache: Cache) {}

  async getCachedDashboardView(orderId: string): Promise<any> {
    const tenantId = tenantStorage.getStore();
    // Strict isolation caching structural mask pattern
    const cacheKey = `tenant:${tenantId}:order:${orderId}:dashboard`;

    const cachedData = await this.redisCache.get(cacheKey);
    if (cachedData) {
      return JSON.parse(cachedData as string);
    }

    // Fallback: Query MongoDB Tenant Isolated Read Collection if cache miss happens
    const freshView = await this.fetchFromMongoModel(tenantId, orderId);
    
    await this.redisCache.set(cacheKey, JSON.stringify(freshView), 10000); // 10s TTL
    return freshView;
  }

  private async fetchFromMongoModel(tenantId: string, orderId: string) {
    // Logic filtering query by { tenantId, orderId } goes here
    return { orderId, tenantId, status: 'COOKING_IN_PROGRESS' };
  }
}
